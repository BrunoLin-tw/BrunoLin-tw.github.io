最近 OpenAI 提到 Codex 已經開始支援 subagents。 <https://developers.openai.com/codex/subagents>
終於也跟上Claude Code的腳步了!

![Codex Subagents 上線：讓多個代理同時幫你工作](/assets/images/posts/2026-03-20-codex-subagents-hero.png)

表面上看，這像是 AI coding 工具又多了一個新功能；但往深一點看，這件事真正值得注意的，不是 AI 更會寫程式，而是 AI 開始更會分工。

過去大家對 AI coding 的期待，大多停留在補 code、修 bug、寫測試、改註解。這些都很實用，也確實提升了效率，但本質上還是在原本的工作流裡，把其中幾個步驟交給 AI 代勞。

subagents 不一樣。它的價值不在於把某一段 code 寫得更快，而在於讓整個開發流程更像多工協作。這代表 AI coding 的重點，正在從「幫你寫」往「幫你拆任務、幫你分工」移動。

## 從單一 copilot 到多個 agent 分工

如果只有一個 AI assistant，再強也常常只能跟著主線任務一路往前走。你問一個問題，它回答一個問題；你貼一段 code，它針對那段 code 給建議。這種模式像是一個很強的副駕，但整個方向盤還是你自己一直抓著。

subagents 更像是你手上開始有多個可以分派任務的助手：

- 一個去看某個 bug 的可能成因
- 一個去查文件或 API 行為
- 一個去整理某個 module 的重構方向
- 一個去補測試或驗證邊界條件

真正的改變，不是單一 agent 更聰明，而是原本混在同一個 context 裡的工作，終於可以拆開來處理。

## 真正的價值是減少複雜度

我覺得 subagents 最重要的價值，甚至不是平行處理本身，而是它有機會減少 context overload。

現在很多 AI coding 工具的問題，不是模型不夠強，而是主對話很容易越用越肥。需求、限制、架構背景、調查過程、失敗方向全部混在一起，最後主 context 變得很重，連人自己都不想重看一遍。

subagents 的好處，是你可以把局部任務切出去處理，讓主線保留在真正重要的決策上。也就是說，AI 不只是幫你產生內容，而是開始幫你管理工作流的複雜度。

在真實開發裡，很多時候最耗成本的不是「寫出那幾行 code」，而是來回切換上下文、追查問題、確認假設、重建思路。誰能幫你降低這些摩擦，誰就更接近真正的 productivity tool。

## 軟體工程師的角色也在變

這也是我覺得 subagents 最值得注意的一點：它不只是改變工具，也在改變軟體工程師的角色。

當 AI 開始可以分頭處理局部任務，工程師的工作就不再只是自己一路寫到底，而是要更像一個安排 agents 分工合作的協調者。

你要決定任務怎麼拆、哪些事情可以平行、哪些結果需要人工判斷、哪些結論可以收斂回主線。換句話說，AI 幫你分工，不代表你可以放棄思考；反而更要求你對問題有結構化理解。

從這個角度看，AI coding 的下一步，未必只是模型誰更會寫，而是誰更會 orchestration。真正拉開差距的，可能不是補完能力，而是能不能幫開發者分配任務、管理複雜度、維持主線清晰。

## 結論

所以如果要我用一句話總結，我會說：Codex subagents 值得注意，不是因為它讓 AI 更像一個會寫程式的人，而是它讓 AI coding 工具更像一個能參與分工的團隊成員。

從單一 copilot，到多個 agent 分工協作，這看起來只是功能增加，但背後反映的其實是同一件事：AI coding 正在從「協助寫 code」走向「協助組織工作」。

如果這條路走得通，那麼軟體工程師的角色，也會慢慢從單純的執行者，轉向更像 orchestrator——負責定義問題、安排分工、收斂結果，讓整個開發流程更有效率。

這樣的 AI，才比較像是工作流工具，而不只是進階版 autocomplete。
