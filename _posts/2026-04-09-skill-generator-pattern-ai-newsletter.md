我最近做了一個自己很有感的工作流優化。

原本我的 AI 每日電子報，是靠 cron job 直接丟一大段 prompt 去跑。裡面塞滿選題規則、去重邏輯、輸出格式、ledger 更新方式、發布流程，能跑是能跑，但老實說不太優雅，也不好維護。

後來我把這整段流程改寫成一個 **generator pattern 的 skill**，把真正重要的規則和模板全部收進 skill 裡。結果最後留在提示詞裡的，只剩一句話：

> 幫我產生 AI 每日電子報。

然後就真的可以跑完。

![把冗長提示詞封裝成 Skill 膠囊的工作流示意圖](/assets/images/posts/2026-04-09-13-51-00-skill-capsule.png)

## 背景

這次調整，表面上看是把 prompt 變短，但我覺得本質上比較像是把「一次性的提示詞技巧」，改成「可以維護的工作流結構」。

我參考的是 Google 提出的 **5 大 Agent Skill 設計模式**，其中我自己最有感的是 **generator pattern**。簡單講就是：

- 使用者只講意圖
- 複雜規則收進 skill
- 輸出格式和流程規範也定義在 skill 裡
- agent 先讀 skill，再執行任務

這樣做的好處很直接，prompt 不用每次重寫，規則也不會散落在各種 cron 設定裡。

## 原本做法的問題

一開始直接把所有流程寫進 cron prompt，最大的問題有幾個。

第一個是 **太長**。  
長到每次想改一個小地方，都要重新讀完整段，浪費時間又容易漏。

第二個是 **規則和意圖混在一起**。  
我只是想要「產生 AI 每日電子報」，但 prompt 裡面其實同時還背著資料結構、路徑規則、輸出模板、發文條件，整個層次是亂的。

第三個是 **錯誤很難收斂**。  
一旦出問題，會搞不清楚到底是 prompt 寫得不好、流程設計有洞、路徑設定錯了，還是輸出格式不對。

這種東西一開始還能忍，但只要 workflow 會長期跑，就會開始覺得很煩。

## 我怎麼改成 Skill

我後來做的事情很單純，就是把原本埋在 prompt 裡的東西，一項一項抽出來，收進 skill。

例如：

- 先確認 Asia/Taipei 當前日期
- 去哪裡找新聞候選
- 怎麼抓全文與篩選來源
- 怎麼跟過去 7 天 ledger 去重
- 電子報 Markdown 要長什麼樣子
- ledger JSON 要怎麼 append
- publish script 什麼時候才能跑
- 發布成功後 Telegram 要怎麼通知

這些其實都不是「使用者意圖」，而是「工作流知識」。

當這些東西都進 skill 之後，真正的 prompt 就只需要保留意圖本身：

```text
幫我產生 AI 每日電子報
```

這種感覺很像把複雜度往系統內部收，而不是留在每次互動時重複攤開。

## 為什麼這樣比較舒服

我自己用下來，最有感的是三件事。

### 1. Prompt 終於回到它該有的樣子

以前的 prompt 比較像規格書。  
現在的 prompt 比較像命令本身。

我只要講我要什麼，不用每次把怎麼做一起背出來。

### 2. 出錯時比較知道該修哪裡

這點超重要。

以前出錯會想先補 prompt，但補久了常常只是把問題越補越亂。現在我會優先看 skill 定義、流程規則、路徑配置，修正點變得清楚很多。

也就是說，**不是一直 patch prompt，而是去修 workflow 本體**。

### 3. 這比較像真正能長期維護的 agent workflow

如果只是臨時玩一下，長 prompt 無所謂。  
但如果是每天跑、每週跑、會持續用的工作流，我現在會很傾向把它 skill 化。

因為這樣才有辦法：

- 重用
- 擴充
- 追錯
- 逐步修正
- 讓 agent 之後自己讀規則做事

## 中間當然還是有踩坑

實作過程中不是零失誤，還是有出現幾個小錯。

像是：

- skill 的參考路徑一開始對錯地方
- project 路徑跟 skill 目錄有混到
- 搜尋來源品質不穩，要回頭補抓一手來源
- 去重規則要再收斂，不然昨天類似新聞今天又可能進來

但這裡反而是我覺得整個流程最有意思的地方。

因為我不需要自己跳下去逐行 debug。  
我可以直接讓 agent 去：

1. 檢查問題
2. 解釋原因
3. 提出修正方案
4. 然後我只負責決定要採哪個方案

這種分工我很喜歡。  
人不是消失，而是從處理瑣碎細節，變成負責判斷方向。

## 這次最大的體感改變

以前我在維護的是一段 prompt。  
現在我在維護的是一套 skill-based workflow。

這兩個看起來很像，但本質差很多。

前者比較像技巧。  
後者比較像系統。

一旦這個模式跑順，後面很多事情其實都能照樣套：

- 每日摘要
- 固定監控任務
- 研究整理
- 報表生成
- 部落格草稿產生
- 定期通知流程

所以我現在越來越覺得，重點不是把 prompt 寫得多神，而是讓 agent 背後有沒有一個穩定、可維護、可演進的骨架。

## 結論

如果你現在也有一些固定讓 agent 跑的工作，我真的會建議你試一次：

**把長 prompt 拆掉，改成 skill。**

把流程規則、資料結構、輸出模板、錯誤處理收進 skill，讓提示詞只保留意圖。

最後你得到的好處通常不是只有 prompt 變短，而是整個 workflow 會變得：

- 更乾淨
- 更穩定
- 更容易修
- 更像真正可以長期用的系統

對我來說，這次最爽的結果就是這一句：

> 幫我產生 AI 每日電子報。

就這樣。剩下的交給 skill。
