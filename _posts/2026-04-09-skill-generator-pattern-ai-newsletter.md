以前我把 AI 每日電子報工作流直接塞在 cron job 的提示詞裡，內容又長又碎，包含選題規則、輸出格式、去重邏輯、發布流程，改一次就很痛苦。

最近我把這套流程重構成 **generator pattern 的 skill**，把工作規範、輸出模板、資料路徑、發布順序全部收進 skill 裡。結果很直接，原本冗長的提示詞，最後只剩一句：

> 幫我產生 AI 每日電子報。

真的就搞定。

![把冗長提示詞封裝成 Skill 膠囊的工作流示意圖](/assets/images/posts/2026-04-09-13-51-00-skill-capsule.png)

## 背景

這次調整的核心，不只是把 prompt 變短，而是把「工作流知識」從一次性的提示詞，提升成可維護、可重用、可逐步修正的 skill。

我參考的是 Google 提出的 **5 大 Agent Skill 設計模式**，其中最有感的是 **generator pattern**：

- 使用者提示詞盡量短
- 複雜規則放進 skill
- 輸出格式由 skill 統一定義
- agent 先讀 skill，再執行工作
- 發現問題後，優先修 skill，而不是一直補 prompt

這種做法很像把 prompt engineering 往前推進一步，變成 workflow engineering。

## 原本做法的問題

直接把完整流程寫在 cron prompt 裡，有幾個明顯缺點：

1. **提示詞太長**  
   每次看都累，改也容易漏。

2. **規則跟輸出格式混在一起**  
   選題、去重、寫作格式、發布順序全部纏成一團。

3. **維護成本高**  
   只要改一個小規則，就要重改整段 prompt。

4. **錯誤不容易收斂**  
   出問題時，很難快速看出到底是流程設計錯、資料路徑錯，還是輸出格式錯。

## 我怎麼改成 Skill

我把整個 AI 電子報工作流拆成 skill，裡面明確定義幾件事：

- 先取得 Asia/Taipei 當前日期
- 去哪裡找候選新聞
- 怎麼讀全文與篩選
- 怎麼跟過去 7 天 ledger 去重
- 電子報 Markdown 模板長什麼樣子
- ledger JSON 要怎麼 append
- publish script 什麼時候才能執行
- 成功後 Telegram 通知格式是什麼

也就是說，**流程規範、資料結構、輸出模板、發布規則**，全部集中到 skill 裡。

這樣之後真正呼叫 agent 的提示詞就可以短到只剩一句：

```text
幫我產生 AI 每日電子報
```

短得很扯，但很爽。

## 為什麼這樣比較好

我自己用下來，最大的差別有三個。

### 1. Prompt 變乾淨

以前 prompt 像規格書，現在 prompt 比較像 intent。

我只要說我想做什麼，不用每次重講怎麼做。

### 2. 修正點集中

當流程有 bug，不是去補 prompt，而是回頭修 skill。

這會讓整個系統越用越穩，而不是越補越亂。

### 3. 更接近真正可維護的 agent workflow

如果 workflow 要長期跑，尤其是 cron、排程、定期產出這種場景，把規則藏在 prompt 裡其實很蠢。  
把規則收進 skill，才比較像真正的工程化做法。

## 中間當然不是零失誤

實作過程中還是有小錯，例如：

- skill 裡的參考路徑對錯根目錄
- project 路徑跟 skill 目錄混用
- 搜尋結果品質不穩，需要補抓一手來源
- 去重邏輯要再收斂，避免昨天新聞今天又進來

但這裡反而是我覺得最有價值的地方：

**我不需要自己跳下去逐行 debug。**

我可以直接讓 agent：

1. 檢查發生什麼問題
2. 說明原因
3. 提出可行修正方案
4. 讓我只決定採哪個方案

也就是說，人類不再負責微調每個細節，而是負責做最後決策。

這個分工，我覺得很合理。

## 這次最大的體感改變

以前我在維護一個 prompt。  
現在我是在維護一個 skill-based workflow。

這兩件事看起來很像，其實差很多。

前者偏一次性技巧。  
後者比較像可演進的系統。

一旦 workflow 成形，後面很多類似工作其實都能照這個模式搬：

- 每日摘要
- 研究報告整理
- 定期監控與通知
- 部落格草稿產生
- 專案狀態彙整

重點不是「讓 prompt 更厲害」，而是「讓 agent 有穩定可重用的工作骨架」。

## 結論

如果你現在也在用 agent 跑固定工作流，我很推薦你做一件事：

**把長 prompt 拆掉，改成 skill。**

把流程規範、資料結構、輸出模板、錯誤處理都放進 skill，讓提示詞只保留意圖。

最後你會得到一種很舒服的狀態：

- 指令變短
- 規則變清楚
- 維護變容易
- 出錯也比較好修

對我來說，最終效果就是這樣一句話：

> 幫我產生 AI 每日電子報。

然後，真的搞定。
